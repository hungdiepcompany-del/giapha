# Backup Manifest Integrity Checker

## Status

BACKUP_MANIFEST_INTEGRITY_CHECKER_BASELINE

Phase 24 adds a fixture-only manifest integrity checker for backup automation readiness. It reads only the sample fixture files under `fixtures/backup/`, recomputes checksum data locally and does not read env files, call network/API/DB, upload files, restore data, deploy, push, change schema, or mutate real data.

## Production Baseline

- Production URL remains unchanged.
- No custom domain cutover is performed in this phase.
- No scheduled backup job or cron is created in this phase.
- No real backup file is created from production data.
- No restore action is performed.

## Integrity Goal

The goal is to prove that a backup manifest can be checked against its referenced payload before any real backup automation exists.

The verify command must print `FIXTURE_ONLY`.

## Input Files

- Fixture: `fixtures/backup/sample-family.fixture.json`
- Manifest: `fixtures/backup/sample-family.manifest.fixture.json`

Both files are static sample files generated by `npm run backup:fixture:generate`.

## Manifest Checks

The checker validates:

- `project` is `gia-pha`
- `environment` is `fixture`
- `backup_type` is `sample_fixture`
- `fixture_marker` is `SAMPLE_FIXTURE_ONLY`
- `contains_real_data` is `false`
- `contains_secret` is `false`
- `checksum_algorithm` is `sha256`
- `fixture_file` points to `sample-family.fixture.json`
- `people_count` matches fixture people count
- `relationship_count` matches fixture parent/child edge count
- `fixture_checksum_sha256` matches the actual fixture content

## Fixture Checks

The checker validates:

- Fixture metadata exists.
- Fixture environment is `fixture`.
- Fixture marker is `SAMPLE_FIXTURE_ONLY`.
- Fixture uses only sample names.
- Fixture has 3-5 fake people.
- Fixture has at least one family.
- Fixture has parent and child relationships.
- Fixture does not contain secret-like strings.

## Verify Command

Run:

```bash
npm run backup:fixture:verify
```

Expected safe output:

```text
[backup:fixture:verify] FIXTURE_ONLY
[backup:fixture:verify] Manifest shape: PASS
[backup:fixture:verify] Fixture shape: PASS
[backup:fixture:verify] Checksum: PASS
[backup:fixture:verify] Secret scan: PASS
[backup:fixture:verify] Result: PASS
```

## Secret/Privacy Guardrails

- Do not read `.env.local`.
- Do not read `.dev.vars`.
- Do not print secret/token/key values.
- Do not include real family data.
- Do not include public download URLs.
- Do not upload fixture files.
- Treat fixture files as local sample data only.

## PASS/FAIL Criteria

PASS when:

- Manifest and fixture files exist.
- Manifest shape is valid.
- Fixture shape is valid.
- Manifest count fields match fixture counts.
- SHA-256 checksum matches fixture content.
- Secret pattern scan passes.
- The command prints `FIXTURE_ONLY`.

FAIL when:

- Fixture or manifest file is missing.
- Checksum does not match.
- Manifest count fields drift from fixture data.
- Fixture is marked as production or real data.
- Secret-like values are detected.
- Script tries to read env files or call external services.

## Phase 24 Boundary

- Do not deploy in Phase 24.
- Do not push.
- Do not change schema.
- Do not create or run migrations.
- Do not mutate real data.
- Do not read `.env.local`.
- Do not read `.dev.vars`.
- Do not create a real scheduled job.
- Do not create/upload a production backup.
- Do not restore production.
- Do not hardcode secret/token/key values.
